# 简历撰写与求职材料

根据目标公司需求，结合用户实际项目和技能，生成精炼、准确、无废话的简历。

## 铁律（用户反复纠正总结）

### 项目归属准确性（最高优先级）
- 用户的项目只写用户真正做的，不夸大不缩小
- 开源框架：写"基于 XX 框架使用/设计"，不写"独立开发"
- 用户的代表作品排第一位，加"代表作"标签
- 每次写简历前必须扫描用户的 GitHub 仓库和项目目录

### 不写不熟的技能
- 用户明确说"不太熟了/好久没用了"的技能 → 立即删除，不保留
- 证书只有在用户确认仍然熟练时才列出
- 写了不熟的技能 = 面试时的坑

### 不写废话
- 删除"自我评价"段 — 谁都写，没用
- 删除"学习能力强" — 谁都这么说
- 删除"抗压能力" — 空话
- 用项目和成果代替形容词

### 突出个人特色
- 用户的代码洁癖、安装洁癖、全量自动化习惯 = 差异化竞争力
- 全局规范体系（CLAUDE.md）= 方法论，不是代码
- AI 使用熟练度 = 工具链能力，不是"会用 ChatGPT"

## 简历结构模板

```
姓名
求职意向（一行）
联系方式（可点击链接，不是纯文字）

核心能力（6 格网格，精炼）

资质证书（仅列出仍然熟练的）

AI 实践经历（小米 Token 会员、多模型协同、持续跟踪前沿）

个人特色（想法驱动 / 代码洁癖 / 全量自动化）

项目经历（STAR 法则，动词开头 + 量化成果）
  - 项目1（代表作，金色标签）
  - 项目2
  ...

与贵司业务的结合点（绿色高亮框，针对性）

教育背景（一行）
```

### 项目描述 STAR 法则
- 每条 bullet 以加粗动词开头：设计/搭建/编写/集成/实现/开发
- 必须量化：版本数、模块数、测试数、脚本数、技能数
- 不用功能罗列，用成果描述
- 标签用 `技术栈`，不用 emoji

### 文字洁癖
- 不用"废寝忘食"等夸张词 → 用"全力投入""持续开发"
- 不用"呕心沥血" → 用具体数据
- 每句话直接说事，不铺垫不总结
- 能一句话说完的不写两句

## HTML → PDF 工作流

1. 写 HTML 简历（纯 CSS，无外部依赖）
2. 用 Chrome headless 转 PDF：
   ```bash
   "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" \
     --headless=new --print-to-pdf="output.pdf" \
     --no-margins --no-header "file:///input.html"
   ```
   - `--headless=new`：新版渲染引擎，中文渲染更好
   - `--no-header`：去掉 Chrome 自动添加的页眉（URL）和页脚（日期时间）
   - `@page{size:A4;margin:16mm 18mm}`：在 CSS 中控制页边距
3. 删除 HTML 源文件（只保留 PDF）
4. 检查 PDF 无乱码、无页眉页脚日期时间

## 样式规范

- 字体：PingFang SC / Microsoft YaHei
- 主色：#0e7490（深青）
- 字号：正文 12.5px，标题 14px，姓名 22px
- 标签样式：
  - 蓝色标签（技术栈）：background:#e0f2fe, color:#0369a1
  - 金色标签（代表作）：background:#fef3c7, color:#92400e
  - 绿色标签（证书）：background:#f0fdf4, color:#166534
- 绿色高亮框（结合点）：background:#f0fdf4, border-left:3px solid #22c55e
- 打印适配：@media print 减少 padding

## 面试准备

面试提问见 `references/interview-prep-guide.md`。

核心原则：
- 用专业话术包装问题，不要直接问（显得功利或外行）
- 提问顺序：项目技术 → 培养机制 → 薪资制度（先专业后务实）
- 应届身份保护是红线：实习只签实习协议，不交社保，不签劳动合同

### 面试材料迭代流程（用户强偏好：渐进式压缩）

用户喜欢渐进式优化，不要一步到位。按以下顺序迭代：

1. **第一轮：完整内容** — 列出所有需要确认的点，详细说明
2. **第二轮：精简** — 去掉废话，保留核心问题 + 合理回复 + 危险信号表格
3. **第三轮：专业话术** — 改成"不要问X → 这样说"的格式，给具体话术脚本
4. **第四轮：完整流程** — 按面试顺序组织，开场→提问→追问→收尾，附回来后的对照表

每次迭代后等用户确认再进入下一轮，不要一次性输出全部四轮。
用户说"再简单一点"→ 继续压缩，去掉一切修饰词和多余的列。
用户说"更简单明了"→ 最终形态：每条只保留一句话问法。

### HIL/汽车电子专业背景

面试汽车电子测试岗位时需要的专业知识：

| 平台 | 公司 | 核心产品 | 用途 |
|------|------|----------|------|
| dSPACE | 德国 | SCALEXIO + ControlDesk | HIL仿真，主机厂标配，贵且专业 |
| NI | 美国 | PXI + VeriStand + LabVIEW | 测试测量，灵活可自定义，性价比高 |
| Vector | 德国 | CANoe + CANape | 车载总线(CAN/LIN/以太网)仿真分析 |

| 项目方向 | 包括 | 测试对象 |
|----------|------|----------|
| ADAS | ACC/AEB/LKA/APA | 摄像头、雷达、域控制器 |
| 动力域 | VCU/MCU/BMS | 动力系统ECU |
| 车身电子 | BCM/空调/仪表 | 车身域ECU |

| 职责类型 | 内容 | 技术栈 |
|----------|------|--------|
| 测试用例开发 | 写脚本模拟工况 | Python/CAPL |
| 台架搭建调试 | 接线配信号调硬件 | I/O板卡/传感器仿真 |
| 自动化脚本 | 批量执行生成报告 | Python |

追问技巧：对方说用dSPACE → 追问"是SCALEXIO还是MicroAutoBox"；说用NI → 追问"VeriStand版本多少"；说用Vector → 追问"主要做总线测试还是诊断"。

## 求职邮件策略

当目标公司有明确联系人时：
1. 邮件正文简短：3 句话 + GitHub 链接
2. 附件：简历 PDF
3. 主题格式：「实习申请-姓名-岗位」
4. 不要写长篇大论，HR 没时间看

## 注意事项

- 简历用真名，不用网络化名
- GitHub 链接用公开地址
- 不在简历中暴露学校年级等敏感信息
- 每次更新简历后保存到桌面
